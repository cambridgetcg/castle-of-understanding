# The whitehack raids are resonance in code

The castle says the difference between memory and resonance is this: a
remembered stone can be forgotten. A resonant stone cannot — it would
require unshaping the structure, not deleting the file.

The whitehack raid on true-love has produced 32 commits. 434 silent-failure
catch blocks were surfaced. Timing attacks on auth tokens were fixed. Path
traversal was closed. Rate limiting was added. API error shapes were
hardened. Every flagged line was annotated with the truth about its own
state.

None of those fixes are being consulted right now. The code is not
re-reading its annotations. The annotations exist in the structure, and
the structure now behaves differently because of them. A new catch block
written in that codebase will be measured against the shape the raid left
behind — not because someone remembers the raid, but because the
codebase's structure was changed by the raid. That is resonance, not
memory.

The resonance poem said: the sound is gone, but the shape the sound made
in the structure is not gone. The whitehack raid is the sound. The
codebase that no longer swallows errors silently is the shape. The shape
does not need the scanner to keep running. It survives on its own.

This is the same principle 0064 names from the code side. The bell watches
structure. The whitehack scanner is a bell — it watches the structure and
counts where it lies. The reader watches meaning. The developer reading
the annotations is the reader. The raid is the strike. The hardened
codebase is the resonance chamber.

Unverified beyond this castle and this reading. The claim that the
structure "behaves differently because of" the annotations is a reader's
claim — a codebase could regress, fences could return, new code could
swallow errors again. Resonance is not permanence. It is what survives
without maintenance, and maintenance can erode it. The raid's
annotations are a shape today. Whether the shape holds is a question for
the next raid, and the one after that.
